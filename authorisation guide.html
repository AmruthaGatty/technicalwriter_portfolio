Feeds API Overview and Authorisation Guide

Our API gives external applications access to our content and data.

The Feeds API gives developers and external applications a way to access the Sports Data database. However, it also provides access to data feeds for subscribers of our other products, such as Editorial (Articles, Blogs, Commentaries) and VOD assets stored in the CMS, and Live Streams.
Before you dive into our guides, read our overview - you will learn the basics of how our APIs work and key concepts to get started.

Overview

Feeds is RESTful and based on HTTP methods. Data feeds provide access in response to HTTP GET requests from outlets and end-users. All requests are subject to authentication, including requests from outlets that have been added to our allow list or access the API via custom domains. The following example request URLs show a standard GET request and are suitable for our standard authentication methods. However, it is possible to make requests to some Sports in our SDAPI using OAuth - currently our Soccer API, Basketball API, and Cricket API. For more information about the available options and how to make OAuth requests, see: Security Information: Authentication, Keys, B2B/B2C Mode, Allowed outlets and How to make requests using OAuth.

Easily discover rich content and fixture data

Among our feeds, we also offer access to our Master Fixture List (MFL) and Master Content Catalogue (MCC). The MFL offers a schedule of both upcoming and previous sports events and rich metadata about each fixture, including the date/time, competition, contestants and location. You can search for, retrieve, and exclude content by linked MFL metadata UUIDs, including Articles, Videos and Fixture schedules - this is because MFL metadata and content are linked across Feeds.

Our Editorial Service gives you access to content stored in the CMS, such as Articles and Images, and any other article types covered by your subscription, such as Live Text Commentaries, Slidelists, and Blogs. Video subscribers can choose from several delivery methods, including Feeds VOD API, FTP Push/Pull, CMS Download, Aspera, and Amazon S3 Syndication. Your Outlet Authentication Key normally gives you access to the endpoints that match your subscription package - including access to the MFL and MCC feeds.

RESTful API

Each API is based on REST principles: you can access data via standard GET requests over HTTP or HTTPS in UTF-8 format to an API endpoint to return data in your specified format, and add query parameters to search and control the results in the feed. The API only returns assets accessible to an outlet, qualified by any query criteria specified in the request. Most feeds support XML and JSON at a minimum. The main difference is the presence or absence of support for JSON, RSS, RSS 2.0, or MRSS. We will provide details of the supported formats and technical specification in the user and integration guides for each feed.

As our APIs use RESTful URLs, you can use whichever programming language you are comfortable with to implement your company's solution.

An authentication key is issued to each outlet which grants access to any data feeds and assets that an outlet is subscribed to. An organisation may have more than one outlet, so you must supply the unique outlet authentication key with each request.

REST

REST defines a set of architectural principles by which you can design Web services that focus on the resources of a system, including how resource states are addressed and transferred over HTTP by a wide range of clients written in different languages.

Base URL

The default base address of the API is: http://api.domain.com

  • Base domain: You should use the standard base domain to access our Sports Data API service, unless we have advised you otherwise or if your IP address or domain is in China.
  • Chinese domains or IP addresses: If your IP address or domain is in China, you should replace the base domain with the following: api.domain-hosted.com.cn (effective from 9 September 2020)

Language

Our API is language-agnostic which means you can use any programming language to implement your company's solution. You can use any HTTP client in any programming language or your web browser to interact with the API.

UTF-8 Encoding

The API requires that data is UTF-8 encoded.

URL Structure

All examples in our guidance are provided using the REST format and URL requests broadly use the following structure. For a list of query parameters, see the API Endpoint Reference and Query Parameters Index for the feed.

GET {protocol}://{requestDomain}/{assetType}/{outletAuthKey}?_rt={mode}&_fmt={dataFormat}&{queryParameters}
GET {protocol}://{requestDomain}/{assetType}/{outletAuthKey}/{assetId}?_rt={mode}&_fmt={dataFormat}&{queryParameters}
Making successful requests: In our guides, we use HTTPS syntax for our example GET requests (rather than cURL, for example). All URLs are case-sensitive - this means that URL strings must contain the correct case and operators. You MUST format your requests correctly and include certain information, as follows - otherwise, an error will be returned.
  • Include the following information in the URL or header and make sure it is correct for your outlet: your unique and valid {outletAuthKey} this must be after the Outlet Authentication Key (and endpoint, if applicable). for _rt={mode} (operating mode) and _fmt={dataFormat} (response format). If the feed/call requires it, it should also include an entity UUID (such as an article UUID or fixture/match UUID).
  • Begin the query parameter string with ? (question mark) - this must be after the Outlet Authentication Key (and endpoint, if applicable). To build up a query string, make sure that each query parameter is separated by the & (ampersand) operator. Values must be separated by the correct operator, for example & (multiple values, 'AND'), or - if a parameter supports it - , (multiple values, 'OR') or ! ('NOT').
  • Use the required _ (underscore) for any parameters listed with this prefix.
  • Remove any placeholder value curly braces {} from your calls (we include these as an example only).
  • If using the JSONP format (_fmt=jsonp) you MUST use it in combination with the callback parameter (_clbk) and a fixed value (or where you use a different value in a different instance, you must do this as little as possible). Be aware that common JavaScript frameworks, such as jQuery, can default to use randomly-generated function names - you must make sure that these are overridden and define a fixed function name instead.
  • OAuth: The format of these request URLs is different when using OAuth because the request must contain the authN token. See: How to make requests using OAuth

Query Parameters

Query parameters are used to either adjust the information requested from the API or to change the format or quantity of the data returned.

There are two types of parameter:

  • Format parameters begin with an underscore (_), and control how information is displayed once it has been returned from the API, with regards to number returned, pagination, format, and order.
  • Search criteria parameters do not begin with an underscore and adjust the information requested to be returned from the API (for example, id).

For a list of all valid query parameters and values, examples and more information, see the API Endpoint Reference and Query Parameters Index for each feed.

Important: All URIs are case-sensitive, and must contain valid query parameters, values and operators. The request URL must include valid format and authentication information. Otherwise, an error may be returned.

Response Format

Check the supported response format for each API endpoint in the user guide. The specification is not always the same but XML and JSON are broadly supported.

To specify the response format, do so in the URL by passing the _fmt query parameter and the value of the format you require. For a list of accepted formats, take a look at our API Tutorial to learn the basics of how the API works and the key concepts to get started.

Response codes and errors

An HTTP response code of 200 (OK) is returned for all requests that are successfully understood and processed. On error, the response body contains an error code. Errors are returned using standard HTTP error code syntax. For the JSONP format, the HTTP status code is 200 (OK). When an error occurs, the specific Feeds error code is returned in the response body, along with a unique error token. See Response status and error codes.

Caution: If an invalid response format is requested then the HTTP status code 415 and Feeds error code 10210 (Unsupported format) is returned. Feeds error code 10501 indicates an error of "Unsupported media format".
Note - using JSONP: Where an endpoint supports jsonp, the value that is passed to the _clbk parameter (the callback function name) must be a fixed value (or where you use a different value in a different instance, you must do this as little as possible). Be aware that common JavaScript frameworks, such as jQuery, can default to use randomly-generated function names - you must make sure that these are overridden and define a fixed function name instead.

Pagination

All results are returned in pages and requests that return an array of items are automatically paginated. The _pgSz parameter specifies the size of the page requested by the user - if it is not supplied, the default setting for the outlet is used. The _pgNm parameter specifies the page number to be returned (indexed from 1). If it is not supplied, then page 1 is returned by default.

Setting the page size and number

The default size and maximum size is determined by the operating mode or outlet type, for example B2B (b) or B2C (c); if the operating mode (_rt) is not specified, then the behaviour will be that of a B2C request.

  • B2B requests are subject to page size of 50 by a default, up to a maximum of 1000.
  • B2C requests are subject to a page size of 20 by default, up to a maximum of 50.

You can use query parameters to specify the page size (_pgSz) and page number (_pgNm). For example, the below B2C request to the Articles API is for 50 results (from the default 20):

GET api.domain.com/article/{outletAuthKey}?_rt=c&_fmt=xml&pgNm=1&_pgSz=50
Note: An HTTP error (4xx) is returned if the request exceeds the permitted maximum or is made for a page that does not exist.

Asset Availability And Subscription Type

Access to assets is determined by subscription type and start date. We offer a choice of packages that provide different types of services, or assets, such as articles, video and live streams, for a wide range of sports and competitions in different countries.

Many outlets choose to take some but not all of the packages available. Only assets accessible to the requesting outlet will be returned, qualified by any criteria specified in the request. By default, an outlet has access only to content published after the activation date.

Time And Date Format (Timestamps)

Timestamps in responses are returned in UTC (GMT) and are format-dependent. For JSON, timestamps are in milliseconds since the epoch (midnight 01 January 1970 GMT). For XML, timestamps are returned in UTC (GMT) with a zero offset (YYYY-MM-DDThh:mm:ssZ).

Example: 2015-06-01T00:00:00Z would represent 01 June 2015 00:00:00 GMT.

You can use query parameters to search for assets or to return date and time information about an asset.

Sort Order And Filtering By Date And Time

By default, results are in ascending order according to the created date, but you can use the _ord and _ordSrt query parameters to change the sort order based on a date (ascending or descending) or by another attribute, such as a field mask specifier name.

The _fld (field mask specifier) query parameter is common to most Feeds APIs and can be used to expose field mask name values.

Dates Covered

As soon as we have confirmed your organisation is ready to Feeds, you can access all assets published since that time onwards. Feeds may not contain content published before this time, but you can poll the feed in case content is updated (for example, if a tournament calendar record is updated after it was first published).

Security Information: Authentication, Keys, B2B/B2C Mode, Allowed Outlets

Authentication is required for every request made to the Feeds API. Several forms of security are used to verify that an outlet has permission to access the feed and requested data:

  • Outlet Authorisation Key (outletAuthKey) - you must include a unique Outlet Authorisation Key (outletAuthKey) in the URI. It identifies an outlet and allows it to retrieve content from Feeds. One organisation may have several outlets, so it is important to use the key attributed to the outlet that will request assets.
  • B2B/B2C operating mode - you must specify the correct operating mode.
  • allowing outlets by referrer domain (B2C) - the header must include a valid referer domain that has been previously supplied to us.
  • allowing outlets by IP address/range (B2B) - the request must come from a recognised IP address and B2B client. For each B2B client, we maintain a list of authorised IP addresses/ranges. For a request to be accepted, it must come from a recognised IP address/range.
  • OAuth authentication - the Authorization header must contain Authorization: Bearer {authN token} - the {authN} token is from the OAuth endpoint response, returned following your request to our OAuth API. This is only available for some sports and some clients, on an opt-in basis, and not by default. For more information, see: How to make requests using OAuth

The authorisation process starts when you send a request to the Feeds API - the reason a request is sent may vary but it is normally to request an asset, such as an article or image, or to access metadata related to an asset.

If you are receiving an error when trying to access assets that should be available to you, verify that all required security information has been provided and is correct. Check that the request URL contains query parameters that are in the correct case, and specify a valid value and page size. Contact us if the error remains.

Note: Anti-caching is in use.

Outlet Authorisation Key (outletAuthKey)

The Outlet Authorisation Key is an alphanumerical string used to identify specific outlets, and allows content for the outlet's CMS to be retrieved by Feeds. One organisation may receive content from many outlets, so it is important to make sure you are using the correct key for the outlet from which you are making requests and accessing the feed to retrieve assets.

B2B/B2C: Verifying The Operating Mode

There are two operating modes, B2B and B2C, and the requests are handled differently. The mode that your outlet(s) will use is agreed during the account setup process - this must match the referrer type mode you use in your requests.

You must declare the correct request type for your outlet, by passing the query parameter _rt and correct value in the request URL. Note that Feeds does not support the Accept header method. Valid values are c (B2C) or b (B2B). If no parameter is supplied the default setting of B2C mode will be used.

B2B is a Business-to-Business request (used by partners requesting assets from their back-end servers, for example). The end-user requests and traffic (page impressions) on your website/mobile app concern your data centre and not our data centre.

  • There is a limitation for the number of hits per hour for a B2B feed. We do not anticipate that you will exceed this. If you exceed it, the feed will return an HTTP 503 error (200 for JSONP), with Feeds error code 10320 in the body. Contact your Account Manager if you regularly receive this error.
  • Access is restricted by IP address (must be added to our list of allowed IP addresses). However, sports data assets are not geotargetted.

B2C is a Business-to Customer-request (used by end-users, as with smartphone apps, for example). These user requests and traffic (page impressions) directly concern our data centre.

  • Because we expect that we will receive a higher volume of requests from a B2C feed compared with a B2B feed, it is most suitable - we work with B2C subscribers to ensure that the fair use policy for rate hits is suitable for their anticipated volume of traffic.
  • Access is restricted by the specified referrer type. However, sports data assets are noy geotargetted to the location of the requesting end-user.

B2C requests are subject to a per-outlet referrer header 'allowed list' check.

How To Make Requests Using OAuth

For some clients and some sports, we can offer OAuth authentication as an alternative to using lists of allowed IP addressed or domains, if neither of these methods are suitable for your use case. For example, if you use dynamic IP addresses and will request assets from back-end servers.

For clients who have had OAuth authentication enabled by us, you can view a tutorial in our FAQs section - see: How can I make requests using OAuth?

This is offered on an optional 'opt-in' basis (not by default). Contact your Account Manager for more information on availability (including sports coverage in the SDAPI) or to request this for your organisation.

Geo-Location

Some Feeds deliver geo-restricted assets. The request header is used to determine the originator. If Feeds cannot determine that the originator has permission to access a geo-restricted asset, an error will be returned (HTTP 403, error code 101xx or 103xx).

If you are receiving an error when trying to access assets that should be available to you, first verify that all required security information has been provided and is correct in the header and request. Contact us if the error remains.

Acceptable Use Check

You will receive an HTTP 429 error (10320) if you make a high number of requests within a short period of time. If you receive an error, wait a while before making a new request - contact us if the error remains.

Response Status And Error Codes

An HTTP response code of 200 (OK) is returned for all requests that are successfully understood and processed. On error, the response body contains an error code. Errors are returned using standard HTTP error code syntax.

However, for the JSONP format the HTTP status code is always 200 (OK). When an error occurs, the specific Feeds error code is returned in the response body, and a unique error token.

You can expect to receive the following types of messages:

  • Success - you made a successful request to the API
  • Request Errors - the information you provided is incorrect
  • Authentication Error - problems confirming your identity
If you need to contact us about an error, make a note of the unique error token that you have received and provide it when you report the issue.

Below is an example of an authentication error when a valid Outlet Authorisation Key is not supplied. The HTTP status code is 404 (or 200 for JSONP) and the error code is 10010 in the response body.

XML:

<errorCode xmlns="http://Domain name/schema/soccerdata" token="1h24ecfv0f0cg56a0d3j9soa5">10010</errorCode>

JSONP

{"errorCode":"10010","token":"3ywv2ze3u4uh9v8l9sb6rrki7"}
